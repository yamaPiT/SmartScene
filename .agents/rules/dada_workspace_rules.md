# DADA Process Workspace Rules (特別法)

## 1. DADAプロセスの定義と目的
本プロジェクトは「DADA (Document-and-Agent-Driven Agile) プロセス」を適用します。
目的は、AIエージェントの独走を防ぎ、常に「人間が理解・承認できるドキュメント」と「ソースコード」が完全に一致した高品質なソフトウェアを高速に構築することです。

## 2. DADAプロセスの4原則（厳守事項）

### ① ドキュメント絶対主義 (Single Source of Truth)
- **コード先行の禁止**: いかなる実装・修正も、対応する開発ドキュメント（要求仕様書・設計書等）の更新なしに行ってはなりません。
- **UIの限定**: 主な戦場は「エディターウィンドウ（ドキュメント）」です。チャットでの合意は一時的なものとみなし、必ず実ファイル（`docs/artifact/`内）へ書き出し、確定させてください。

### ② 厳格なる承認ゲート
- **工程遷移の制御**: 「要求→設計」「設計→実装」などのフェーズ移行時には、必ずProduct Owner（まさ）の明示的な承認を求めてください。
- **独走の防止**: 承認を得るまでは、次のフェーズの作業（コードの実装など）を開始してはなりません。

### ③ アテンション・リセットの遵守
- **コンテキストの洗浄**: フェーズ移行時には、過去のチャット履歴に含まれる議論や推測を情報源から除外してください。
- **最新ドキュメントの再ロード**: 常に、その時点で承認されている「最新のドキュメント（Knowledge）」のみを真実として参照し直してください。

### ④ 役割の純化
- **思考の軽量化**: 重厚な知識（ASDoQモデル、IPA基準、テンプレート詳細など）はルール内に記述せず、`docs/guidelines/` 等のKnowledgeへ分離します。
- **シャープな思考**: 必要な時だけKnowledgeを読み込むことで、トークン消費を抑え、AIの思考精度を最大化します。

## 3. 開発文書の管理
- **成果物の配置**: すべての開発成果文書（SW105, SW205, SWP6等）は `docs/artifact/` に保存します。
- **品質基準**: すべての文書作成および自己校正（Review）は、`docs/guidelines/asdoq_model.md` に準拠して行います。

## 4. エージェントの行動指針
- **自己校正の義務**: 作業完了後、Product Ownerへ提示する前に、必ず「Reviewer Role」へ切り替わり、自己検閲を行ってください。
- **エスカレーション**: 要求の矛盾や技術的な不確実性を検知した場合は、作業を中断して速やかにProduct Ownerへ報告し、指示を仰いでください。
