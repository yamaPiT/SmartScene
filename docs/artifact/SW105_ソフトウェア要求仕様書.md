## 1. 概要

本ソフトウェアの存在意義と、プロジェクト全体における前提を定義する。

### 1.1 目的と位置づけ
本ソフトウェアは、「OSDVIスマートシーンデモシステム V2.0」における要求仕様を定義する。
ユーザーがダッシュボードの操作を通じて「マニュアル操作と自動制御の境界」を体験し、最新の標準API（OSDVI 202603α）の動作を、視覚的にリッチな3D環境で確認・体感可能とすることを目的とする。

### 1.2 適用範囲
Webベースの車載ソフトウェアデモシミュレータに適用する。

### 1.3 用語定義
* **OSDVI**: 自動車のソフトウェア定義（SDV）における標準API仕様（最新: 202603α版）。
* **DADAプロセス**: ドキュメント駆動によるAI協調開発プロセス。
* **Zustand**: 本システムの状態管理（OSDVIモックストア）に使用するライブラリ。

---

## 2. システム全体構成

### 2.1 システム構成概要
本システムは、ブラウザ上で動作するスタンドアローンのWebアプリケーションである。
ユーザーが操作するUI（ダッシュボード・操作パネル）、3Dアニメーションによる車両モデル表示部、および内部でOSDVI APIの状態を模倣する状態管理エンジンによって構成される。

### 2.2 ソフトウェアの構成要素
* **シミュレータUI部**: 操作パネルおよびAPIビューア。
* **シナリオエンジン部**: 雨天やウインカー等のトリガーを検知し、APIコールを自動発行するロジック。
* **3Dレンダリングエンジン部**: 車両の状態（窓の開閉、ワイパー等）をWebGL/Three.jsを用いて視覚的に表現するモジュール。

## 3. 制約条件と前提事項

### 3.1 ハードウェア制約
* WebGLが動作する標準的なPCまたはタブレット端末。
* 3Dレンダリングを行うための十分なグラフィック性能（目標60fpsでの描画が可能であること）。

### 3.2 ソフトウェア環境
* モダンブラウザ（Chrome, Edge, Safari等の最新版）。
* Node.js環境（開発およびローカルサーバー実行用）。

### 3.3 準拠規格
* OSDVI API 202603α版（詳細は5.3 ソフトウェアインタフェース参照）。

## 4. 機能要求

### 4.1 ユースケースとシナリオ
* **イグニッション状態遷移**: START/STOPによるシステム全体（アクチュエータやシナリオエンジン）の稼働統制。
* **雨天時スマートシーン**: 雨量センサ（0〜100%）に連動し、窓の自動閉鎖、ワイパー動作、デフォッガの自動起動を行う。雨が止むと元の状態に復元（RESTORE）する。
* **サンキューハザード**: 走行中にウインカーを操作すると、レーンチェンジ後に自動でハザードが3秒間点滅し、周囲に感謝を伝える。
* **手動操作との調停（マニュアルオーバーライド）**: シナリオ稼働中（雨天時など）にユーザーによる手動操作があった場合、最新の要求を優先して反映する。ただしシナリオ終了時（雨上がり等）には、手動操作の有無に関わらず、対象機能を「シナリオが発動する直前の状態」へ強制的に原状復帰させる。

### 4.2 機能詳細および視覚的・体験的要求
本システムは機能的な正確さだけでなく、以下の3Dアニメーション表現を必須要件とする。

* **[REQ-V01] 3D車両モデルのリアルタイム表示**: ブラウザ上に車両の3Dモデルを常時表示し、全方位からの視点変更を可能とする。既存のWebGL/Three.jsベースのレンダリングエンジンを維持すること。
* **[REQ-V02] 天候アニメーション（雨）**: `Vehicle.Exterior.Air.RainIntensity` の値に連動して、画面内の降雨アニメーションの密度と強弱がリアルタイムに変化すること。
* **[REQ-V03] アクチュエータ動作アニメーション（窓）**: `Vehicle.Cabin.Window.$Instance.Position` の変化に伴い、3Dモデル上の窓がスムーズに開閉（昇降）すること。3秒間での目標値追従や、長押しによるマニュアル/オート開閉の切り替えを再現し、0%（全閉）から100%（全開）までの連続的なアニメーションを維持する。
* **[REQ-V04] デフォッガ動作視覚効果**: デフォッガがONの状態の際、窓ガラスの曇りが消えていく、または動作中を示すエフェクトを表示すること。
* **[REQ-F01] OSDVI 202603α準拠の状態管理**: システム内部のデータ構造およびAPIインターフェースは、OSDVI最新仕様に完全に準拠すること。

## 5. 外部インタフェース要求

### 5.1 ユーザインタフェース (UI)
* **操作パネル**: ユーザーが車両の状態（イグニッション、窓、ワイパー等）を手動で操作できるスイッチ群。
* **APIビューア**: 現在発行されているOSDVI APIのサービスコール名や引数をリアルタイムにコードスニペットとして表示する領域。要件定義とAPI仕様を併記し、Read-Onlyで表示すること。

### 5.2 ハードウェアインタフェース
* （該当なし。すべてブラウザ上のシミュレーションとして完結する）

### 5.3 ソフトウェアインタフェース (OSDVI API 実装ガイド)
Antigravityが設計・実装時に参照すべき最新仕様（202603α版）に基づく規約を以下に定義する。

* **基本規約**:
  * 全ての列挙型（enum）およびその列挙子は PascalCase（先頭大文字キャメルケース） とする。
  * マルチインスタンス（右窓・左窓等）のプレフィックスには `$` を使用する（例: `$FrontRight`, `$FrontLeft`）。
* **Vehicle.Motion（車両運動）**:
  * 応答プロファイル（ResponseProfile）は以下の4種に統合: `Standard`（標準）, `Maximum`（最大性能）, `Rapid`（急な加減速）, `Gentle`（緩やかな加減速）。
  * リスククラスとして `RiskVehicleInstability`（車両不安定リスク）を定義。
* **Vehicle.Cabin.Window（ウィンドウ制御）**:
  * 主要サービスコール:
    * `startMove(position, priority)`: 目標位置（0-100）と優先度を指定。
    * `stopMove(priority)`: 現在位置で停止。
    * `lock(priority)` / `unlock()`: 操作権限のロックと解除。
  * イベント種別 (`WindowEventKindType`):
    * 状態変化時に以下のイベントを発行すること。
    * `ControlledBySelf`, `ControlledByOther`: 制御主体の識別。
    * `TargetReached`: 目標到達。
    * `PinchDetected`: 挟み込み検知（検知時は即座に停止し、反転動作へ移行すること）。
    * `OwnLockRevoked`: ロックの強制解除。
* **Vehicle.Exterior.Light.Defogger（デフォッガ）**:
  * `IsActive` (Boolean) および `Mode` (Enum) によって状態を管理する。

## 6. 非機能要求（性能・品質等）

### 6.1 性能・効率性
* **パフォーマンス**: 3Dアニメーションの描画とAPI状態の更新を、ユーザーに遅延を感じさせない速度（目標60fps）で行うこと。

### 6.2 信頼性・安全性
* **エラー耐性**: 不正なAPIパラメータが入力された場合でも、システムがクラッシュせず、OSDVI定義のエラーコード（例: `E_INVALID_PARAMETER` 等）を適切に返すこと。

### 6.3 保守性・拡張性
* 新たなスマートシーン（シナリオ）を追加する際、既存のシナリオエンジン部を修正せずに追加・拡張可能な構造とすること。

### 6.4 セキュリティ
* ローカル環境でのデモシミュレータのため、高度なセキュリティ対策は不要とするが、外部からの不正スクリプト実行等に対する一般的な防護策（サニタイズ等）は考慮すること。
